# 📄 项目需求文档：校园文创与活动物料预订管理系统（CampusMerch v2.0）

| 文档版本 | V1.0 |
|:---:|:---|
| 面向对象 | 用户、管理员 |
| 开发周期 | 14 天（前后端分离） |
| 技术栈 | 后端：Laravel 10/11 | 前端：Vue 3 | 存储：MySQL + 对象存储(OSS) + Excel处理 |
| 文档目的 | 明确系统业务边界、功能清单与技术约束，作为前后端架构设计、接口契约制定与开发验收的唯一依据 |

## 1.  项目背景与目标

**业务痛点**：校园社团/实验室在定制文创（文化衫、徽章、笔记本）或活动物料时，长期依赖微信群接龙与手工表格登记，存在以下问题：

- 商品图文信息分散，学生无法直观查看实物效果。
- 定制图案/设计稿通过微信发送，易丢失、版本混乱。
- 预订清单汇总耗时，导出对账/发货报表格式混乱。
- 库存与预订状态不透明，常出现超订或漏单。

**系统目标**：构建一个轻量级预订管理 Web 应用，实现**商品图文在线化、定制设计云端归档、预订流程线上化、发货清单标准化导出**。本期严格控制核心业务接口数量为 **11 个**，通过提升单接口的数据流转复杂度（事务、文件校验、流式处理、并发控制）完成工程能力进阶。

## 2.  用户角色定义

| 角色 | 核心诉求 | 系统权限 |
|:---|:---|:---|
| **用户（学生/教职工）** | “我想浏览商品详情、提交预订、上传定制图案，随时查看订单进度，导出个人消费明细。” | 浏览商品、提交预订、上传附件（设计稿/凭证）、查看个人订单、导出个人记录 |

## 3.  核心功能清单与接口映射（严格 11 个核心业务接口）

> 📌 注：本期核心业务接口严格控制在 **11 个**（不含 `Login/Register/Logout`），通过提升单接口数据流转与校验复杂度完成能力进阶。

| 模块 | 客户功能描述 | 对应接口（RESTful） | 复杂度提示 |
|:---|:---|:---|:---|
| **商品大厅** | 查看文创/物料列表，支持按分类、状态、价格区间筛选，关键词搜索，分页展示 | `GET /api/products` | 多条件动态查询、软删除过滤、响应字段脱敏、库存预警标签 |
| **商品详情** | 查看单件商品完整信息（规格、库存、定制要求、已售量、封面图） | `GET /api/products/{id}` | 关联查询优化、附件聚合、权限过滤 |
| **提交预订** | 选择商品，填写数量、尺寸/颜色偏好、备注，提交预订单 | `POST /api/orders` | 库存预扣、并发防超订、枚举值校验、状态机初始化 |
| **上传定制稿** | 为已提交订单上传设计稿/图案，系统校验格式大小并返回云端链接 | `POST /api/orders/{id}/design` | OSS 上传、MIME/尺寸白名单校验、文件重命名、元数据落库 |
| **确认收货/核销** | 用户收到实物后点击确认，或管理员扫码/手动核销 | `POST /api/orders/{id}/complete` | 状态终态控制、库存最终扣减、事务完整性、防重复点击 |
| **批量上架** | 管理员上传 Excel，系统自动解析并创建/更新商品池，失败行返回明细 | `POST /api/products/import` | 行级校验、事务批量处理、部分成功策略、错误报告生成 |
| **报表导出** | 按当前筛选条件导出 Excel 报表（含订单明细、定制稿链接、发货地址） | `GET /api/orders/export` | 动态列映射、流式输出防 OOM、大文件分块、OSS 链接有效期处理 |
| **订单审核** | 管理员对定制稿进行通过/驳回，或修改预订数量/状态 | `PUT /api/admin/orders/{id}/review` | 状态流转校验、库存/日志联动、幂等性控制、操作留痕 |
| **我的订单** | 查看个人所有预订记录、当前待发货物品、历史已完成明细 | `GET /api/my-orders` | 多表关联、状态聚合、时间轴排序、分页 |
| **商品维护** | 管理员修改商品基础信息、调整库存、变更分类/上架状态 | `PUT /api/products/{id}` | 乐观锁防覆盖、敏感字段变更日志、关联订单校验 |
| **数据看板** | 查看系统核心指标（今日预订量、待审核数、库存预警商品列表） | `GET /api/admin/stats` | 聚合查询优化、缓存策略、权限拦截 |

>  **接口数量控制**：`11` 个核心业务接口 + `3` 个 Auth 接口 = `14` 个。与上一项目完全对齐，但单接口承载逻辑量提升 2~3 倍。

## 4.  核心业务规则与设计约束

### 4.1 订单状态机（必须严格实现）

```
[待提交 draft] → [已预订 booked] → [定制待审 design_pending] 
      ├─(审核通过)→ [制作中/待发货 ready] → (确认收货)→ [已完成 completed]
      └─(审核驳回)→ [已驳回 rejected] → (可重新上传/修改)
```

**规则**：

- 用户提交预订后，系统自动预扣库存（`reserved_qty +1`），防止超卖。
- 仅 `booked` 状态可上传设计稿；仅 `design_pending` 可被管理员审核。
- 状态为终态（`completed`/`rejected`）后不可逆，操作需记录 `audit_log`。

### 4.2 OSS 文件上传规范（设计稿/封面）

- **支持格式**：`jpg, jpeg, png, pdf, ai, psd`（MIME 白名单双重校验）
- **大小限制**：单文件 ≤ 15MB，设计稿分辨率建议 ≥ 300DPI（前端提示，后端校验宽高）
- **存储路径**：`merch-designs/{order_id}/{timestamp}_{uuid}.{ext}`
- **安全要求**：禁止直接暴露 Bucket 域名；返回后端生成的临时签名链接或代理下载路径；上传成功后立即落库关联订单。
- **生命周期**：订单取消或驳回时，物理文件可保留 7 天后定时清理（本期可简化为逻辑删除标记）。

### 4.3 Excel 导入/导出规范

- **导入模板**：首行为 `name, category, type, spec, price, stock, cover_url, custom_rule`
- **校验规则**：必填项缺失、价格≤0、库存<0、枚举值非法行标记失败。
- **处理策略**：采用 `DB::transaction()` 包裹。**合法数据入库，非法数据跳过并返回明细**。支持 2000 行内 3 秒响应。
- **导出要求**：按当前筛选条件动态生成列；采用流式响应（`Laravel Excel ->download()`）；禁止全量加载到内存；生成包含 OSS 可访问下载链接的发货清单。

## 5. 非功能性需求

| 维度 | 要求 |
|:---|:---|
| **文件安全** | 严格 MIME + 扩展名双重校验；上传路径防目录穿越；OSS 链接带时效签名（默认 2 小时） |
| **数据一致性** | 预订扣库存/导入/核销必须事务包裹；失败时完整回滚，不留脏数据或悬挂库存 |
| **性能基线** | 列表接口 `< 500ms`；导入 500 行 `< 3s`；导出 1000 行内存峰值 `< 128MB`；首屏加载 `< 2s` |
| **错误响应** | 统一 `{ code, message, data, errors? }` 格式；Excel 失败需返回 `[{row: 3, field: "price", reason: "必须为正数"}]` |
| **兼容性** | 支持 Chrome/Edge 最新版；导出文件兼容 Excel 2013+ 与 WPS；基础响应式适配 |

## 6.  系统边界（明确排除项）

- ❌ 在线支付接口/微信/支付宝对接（本期仅做“线下支付+线上核销”模拟）
- ❌ 实时聊天/站内信通知
- ❌ 复杂排期/分片上传/断点续传（单文件 ≤15MB，直传即可）
- ❌ 多角色权限组（RBAC）、数据可视化大屏
- ❌ 单元测试与 CI/CD 流水线
- ❌ 独立部署 MinIO（使用阿里云 OSS 免费额度或本地 S3 兼容服务即可）
